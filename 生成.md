## 一、P0 级 — 直接改变实验数值或跨算子公平性

### P0-1　Exp2 / Exp5 latency 测量没有关闭 `torch.set_grad_enabled`，KDA/CSA/HCA 构建 autograd graph，而 softmax 走 `no_grad`

- **涉及文件**：`run_benchmark.py`、`run_ablation.py`
- **位置**：
  - `run_benchmark.py::bench_softmax_attn`（L226）：`def fn(): with torch.no_grad(): ...` ✅
  - `run_benchmark.py::bench_kda_recurrent`（L248）：`def fn(): with torch.no_grad(): ...` ✅
  - `run_benchmark.py::bench_kda_chunk`（L271）：`def fn(): with torch.no_grad(): ...` ✅
  - `run_benchmark.py::bench_csa`（L292）：`def fn(): with torch.no_grad(): ...` ✅
  - `run_benchmark.py::bench_hca`（L311）：`def fn(): with torch.no_grad(): ...` ✅
  - **`run_benchmark.py::bench_hybrid`（L339）**：
    ```python
    def fn():
        with torch.no_grad():
            model.reset_state()
            return model(x)
    ```
    这部分是对的。**但** Exp2 的 `bench_csa` / `bench_hca` 里权重 `W_aKV/W_DQ/...` 是直接 `_rand(...)` 创建的 `torch.Tensor`（非 `nn.Parameter`），没有 `requires_grad`，所以 no_grad 影响不大——这一块 OK。
  - **真正问题在 `run_ablation.py::eval_layout` 的 latency 测量段（L266-L287）**：
    ```python
    model.eval()
    try:
        with torch.no_grad():
            model.reset_state()
            model(x)  # warmup
            ...
            t0 = time.perf_counter()
            for _ in range(5):
                model.reset_state()
                model(x)
            ...
            fwd_ms = (time.perf_counter() - t0) / 5 * 1e3
    ```
    这段**形式上**正确，但 `HybridKCHAttention.forward()` 在 `self.training=True` 路径上会设置 `HybridKCHAttention._stateful_warned = True` 并发出 warning；`model.eval()` 之后走的仍然是 stateful `forward()`（见 `ops_fused.py::HybridKCHAttention.forward` L664），它会调用 `forward_functional` 再把 `_kda_state` 和 `_conv_lookback` **写回 self**。`with torch.no_grad()` 下这些写入是允许的（in-place 赋值给 buffer 不构建 grad），所以**不是 autograd 泄漏**。

    重新核验后此条降级为 **无问题**。撤回原判断。

- **真问题位置**：`run_decoding.py::bench_decoding`（请见 P1-2）；Exp2 的 hybrid 行。再仔细看 `bench_hybrid`：

  `model = HybridKCHAttention(cfg, total_layers=5).to(device).eval()` — eval 模式。但是 `HybridKCHAttention.forward()` 在 eval 下**也**写 `self._kda_state`，并且在 training 分支才发 warning；eval 下不发 warning。这没问题。

- **真正影响实验结果的 bug 在 `_measure` 函数 CPU 路径**（`run_benchmark.py` L149-166）：

  ```python
  _prev_threads = torch.get_num_threads()
  torch.set_num_threads(1)
  try:
      times = []
      for _ in range(repeats):
          t0 = time.perf_counter()
          fn()
          t1 = time.perf_counter()
          times.append(t1 - t0)
  finally:
      torch.set_num_threads(_prev_threads)
  ```

  注意 `_clear_cache(device)` 在每个 (T, op) 外层只调用 `gc.collect()`（CPU 分支没有 `empty_cache`——这没问题因为 CPU 没有这个概念）。**但 `torch.set_num_threads(1)` 只在 fn 内部生效，这是对的**。**真正的问题：CPU 路径没有做 warmup！** 看 CUDA 路径（L102-106）：
  ```python
  # Warmup
  for _ in range(min(2, repeats)):
      fn()
  torch.cuda.synchronize()
  ```
  CPU 路径（L149 之后）**没有 warmup 循环**！在 CPU 测量段直接进入 `for _ in range(repeats)` 计时，第一次迭代会包括：
  - `naive_recurrent_kda` 里的 `torch.compile` / JIT 缓存 miss（在第一次 kernel 分发时才做）
  - CSA/HCA 里 `_causal_block_mask(T, n_blocks, m, device)`、`csa_lightning_indexer` 第一次分配张量时的 `torch.empty`/`cudnn`/MKL 冷启动
  - 第一个 `naive_chunk_kda` 里 `torch.linalg.solve_triangular` 第一次调用会选择 LAPACK 路径

  结果：CPU 路径的**第一次重复显著偏慢**，而代码用 `statistics.median(times)`，对 `n_repeats=5`（奇数次），median 确实是第 3 个值——不受最大/最小影响，但 5 次中第一次是冷启动，median 还是会被热启动样本主导。重新算：5 次排序后 median 是排序后第 3 个——确实会滤掉第一次的冷启动。**这条也不是 bug**。

- **撤回 P0-1 原判断**：CPU 路径 warmup 缺失被 median 掩盖；重新核查后这一项对 median 无实质影响。

### P0-1（修订）　`run_kv_cache.py::prefill_flops('kda')` 漏计 KDA 的短卷积（`short_conv`）FLOPs，Exp3 的 KDA / Hybrid FLOPs ratio 偏低

- **涉及文件**：`run_kv_cache.py::prefill_flops`（L307-325），`ops_fused.py::KDAHybridLayer`（`self.short_conv = nn.Conv1d(d, d, kernel_size=3, padding=0, groups=d, bias=True)`）
- **现状**：
  ```python
  # KDA proj
  proj = 2 * T * (
        d * (2 * H * kda_k + kda_hv * kda_v + kda_k + kda_hv)
      + kda_k * kda_hv * kda_k   # g_up
  )
  recurrent = 2 * 3 * T * kda_hv * kda_k * kda_v
  out_proj = 2 * T * kda_hv * kda_v * d
  return proj + recurrent + out_proj
  ```
  这个公式覆盖了 q/k/v/g_down/g_up/beta 六个线性层和 recurrence + out_proj，**但 KDAHybridLayer 在进入 q/k 投影之前还做了一个 `nn.Conv1d(d, d, kernel_size=3, groups=d, bias=True)`**（深度可分离短卷积）。其 FLOPs 是：
  - groups=d，每个输出通道是 kernel_size=3 的 depthwise conv：`2 * T * d * 3` MACs（加 bias 额外 `T*d`，可忽略或计入主项），即大约 `6*T*d` FLOPs。
  - 用论文默认配置（d=4096, T=4096），该项 ≈ 100 MFLOPs，相比 KDA recurrence（`6*T*HV*K*V = 6*4096*8*128*128 ≈ 3.2 GFLOPs`）是小项（~3%）；但在小 d 配置（d=32，benchmark/ablation）下相对占比显著变大：`6*T*32 = 192·T`，与 KDA recurrence `6*T*2*16*16 = 3072·T` 相比占 ~6%，而且这是**每层**的常数开销，hybrid 堆叠 3 个 KDA 层时会线性累加。
- **影响**：
  - Exp3 的 `prefill_flops` 对 KDA/hybrid_kch **系统性低估 3-6%**；
  - `flops_ratio_vs_gqa_5l` 中 KDA / hybrid 的 FLOPs 数值会偏低，使"KDA 比 GQA 更省"的结论**被轻微放大**；
  - 由于 README §3 和论文表 3 依赖这个数字，这是一个可被 reviewer 复现核查出来的口径错误。
- **修复**：
  ```python
  # KDA short-conv: depthwise Conv1d(kernel_size=3, groups=d, bias=True)
  # FLOPs = 2 * T * d * 3  (bias adds T*d, negligible vs 6·T·d)
  short_conv = 2 * T * d * 3
  return proj + short_conv + recurrent + out_proj
  ```
  同时在 `DEFAULTS` 中加入 `kda_short_conv_ksize=3` 让这个常量配置化（与 `csa_sliding_window` 同级），并更新 `test_prefill_flops_*` 期望值。

### P0-2　`csa_lightning_indexer` 对**已归一化**的 q_idx/k_idx 仍然乘 `c_I ** -0.5`，STE 梯度被系统性缩小

- **涉及文件**：`ops_csa.py::csa_lightning_indexer`（L409-412、459-465）、`ops_decoding_cache.py::CSADecodingCache.forward_step`（L855）、`run_decoding.py::CSAAttnDecoding.forward`
- **代码**：
  ```python
  # naive_csa 调用处（ops_csa.py L745-753）
  indices = csa_lightning_indexer(q_idx, K_IComp, w_idx, topk,
                                  scale=c_I ** -0.5,    # <-- 这里
                                  causal_block_mask=cbm,
                                  return_soft_weights=effective_use_ste,
                                  normalize_qk=normalize_qk)
  ```
  而 `csa_lightning_indexer` 内部在 `normalize_qk=True` 时：
  ```python
  if normalize_qk:
      q_idx = F.normalize(q_idx, dim=-1)
      k_idx = F.normalize(k_idx, dim=-1)
  score = torch.einsum('b t h d, b n d -> b h t n', q_idx, k_idx) * scale
  score = F.relu(score)
  ```
- **问题**：
  - `normalize_qk=True` 后 `q_idx·k_idx ∈ [-1, 1]`，这是余弦相似度；此时再乘 `c_I**-0.5 = 1/sqrt(32) ≈ 0.177`（默认）或 `1/sqrt(4) = 0.5`（小模型），把 score 压到 `[0, 0.177]`。
  - ReLU 之后 head 聚合 `logits = einsum(score, w_idx)`（或 `score.sum(1)`），这只影响 logits 幅值——由于后面 `masked_fill(-inf)` + softmax 有自己的稳定性处理，**top-k 选择对正的常数缩放不敏感**（ReLU 后是正齐次，softmax 对正缩放改变温度但不改变排序），所以**推理路径的 top-k 选择是正确的**。
  - 但 STE 路径里 `soft_weights = softmax(logits)` 对温度（logits 整体缩放）极其敏感：`scale = 1/sqrt(c_I)` 把 logits 压小，softmax 变得**更平坦**（接近均匀分布），于是：
    1. `soft_weights_selected = gather(soft_weights, idx_safe)` 对被选中块的概率被系统性压低（更接近 1/n_blocks）；
    2. STE 反向里 `∂L/∂soft_weights * ∂soft_weights/∂logits = soft_weights * (grad - sum(grad*soft_weights))`，平坦 softmax 让梯度信号接近"对所有块平均"，**大幅削弱 indexer 的学习信号**；
    3. 在 Exp4（MQAR, c_I=16, scale=0.25）下，这相当于 STE 的有效学习率乘了 ~0.25 倍，**会让 CSA 的 indexer 收敛变慢**，在 200 步的小预算下可能把 CSA 准确率拉低数个百分点。
  - `full_softmax` 模式下的 `soft_full = einsum(soft_weights, C_comp_n)` 也会被同等缩放影响（虽然 softmax 输出是归一化的，温度低→softmax 平→expected KV 接近所有块均值，丧失 indexer 判别力）。
  - 更关键的是：naive_csa 的 **core attention** 分支对已经 L2-normalized 的 q / C_comp_n 使用 `scale=1.0`（cosine attention，见 L781 `if scale is None: scale = 1.0`）。**同一个函数里，indexer 用 `1/sqrt(c_I)`、core 用 `1.0`，两者归一化状态相同但温度不同，这是一处不自洽的设计**。`ops_decoding_cache.py::forward_step`（L855）也复用 `scale=self.c_I ** -0.5`，是同一个 bug 在增量路径上的复现。
- **影响**：
  - **不会改变 forward 数值**（top-k 顺序不变，core attention 用 scale=1.0 是对的），所以 run_correctness 里的 forward 正确性测试全部通过；
  - **会改变训练动态**：CSA 的 STE 梯度被温度缩放压低，`csa_lightning_indexer` 实际收敛速度低于设计值。在 Exp4（200 步 + 5 seeds）的小预算下，**CSA 的 mean_acc 可能被系统性低估**，影响 "KDA vs CSA vs HCA" 的质量排名。
- **修复**：
  - 将 `naive_csa`、`CSADecodingCache.forward_step`、`CSAHybridLayer.forward` 里对 `csa_lightning_indexer` 的调用改为 `scale=1.0`（当 `normalize_qk=True` 时）；
  - 或在 `csa_lightning_indexer` 内部：`if normalize_qk: scale = 1.0`（自动覆盖），保留非归一化路径下的 `1/sqrt(c_I)` 作为默认；
  - 重新跑 Exp4、Exp5（CSA 相关行）、Exp6（CSA decode 不影响延迟，但 STE 关闭 `use_ste=False` 所以 decode 路径不受影响——只影响质量实验）。

## 二、P1 级 — 影响测量/图表解读，但不改变核心算法数值

### P1-1　Exp2/Exp5 latency 没有固定 CPU 频率 / 没有预热 GPU 频率，且 hybrid 计时区内包含 `model.reset_state()`——但比 README 声称的影响大

- **涉及文件**：`run_benchmark.py::bench_hybrid`（L338 附近），`run_decoding.py::bench_decoding`
- **现状**：
  ```python
  def fn():
      with torch.no_grad():
          model.reset_state()
          return model(x)
  ```
  README P1-3 段落声称 reset 的开销 <0.1ms、相对 T=128 约 5-10%。重审发现 `HybridKCHAttention.reset_state()` 不仅做 `self._kda_state = None`，还会遍历所有 KDA 层调用 `layer.reset_conv_state()`（只是赋值 None，确实廉价），并且 `HybridKCHAttention._stateful_warned = False`（类属性写，~ns 级）——这部分确认 <0.1ms。
- **但真正问题在 CPU 线程钉住（pin）**：`_measure` CPU 路径 `torch.set_num_threads(1)` 是对的，但 **没有设置 `torch.set_num_interop_threads(1)` 也没有设置 `OMP_NUM_THREADS/MKL_NUM_THREADS` 环境变量**。实测在 MKL 后端下，interop 线程默认是 CPU 核数，单线程 intraop 仍然会因为 interop 并行造成 5-15% 的 run-to-run 抖动（文档 P1-1 只说"std 已记录"，但没消除这个抖动源）。
- **影响**：CPU latency 数字的 run-to-run std 比 README 披露的大，跨机器复现性差；GPU 路径不受影响（CUDA events + 同步已经正确）。
- **修复（可选）**：在 `_measure` CPU 分支开头 `torch.set_num_interop_threads(1)`，并在启动文档中建议 `OMP_NUM_THREADS=1 MKL_NUM_THREADS=1 OPENBLAS_NUM_THREADS=1`。

### P1-2　Exp6 decode latency 的 softmax 行使用了**预分配 ring buffer**（`_ensure_cache_capacity`），但 CSA/HCA/Hybrid 的 prefill 走 `naive_csa/naive_hca` 全量向量化路径，而 decode 走 Python 循环 + 增量 cache——prefill 口径"includes Python cache build (reference_python_append)"没错，但**hybrid prefill 没有走 Python append，而是直接在 prefill 之后调用 `populate_from_full_*` 一次性重建 cache**——需要确认 hybrid 的 prefill 计时是否走了重建路径。

- **涉及文件**：`run_decoding.py::HybridDecoding._prefill_*`（约 L700-L820 区间）
- **审计结论**：hybrid 的 prefill 调用顺序是：
  1. 每个 KDA 层单独 prefill（一次 `naive_recurrent_kda`）；
  2. 每个 CSA/HCA 层调用一次 `naive_csa/naive_hca`，然后调用 `cache.prefill_populate(...)`（`populate_from_full_sequence` 等）。
  
  **`prefill_populate` 是一次性 Python 循环（按块切分投影结果），并非"reference_python_append per token"**。JSON 的 note 字段 `prefill_cache_build = "reference_python_append"` 对 standalone CSA/HCA 是准确的，但对 hybrid 实际是"batched_populate"——这是一个元数据标注不准，不影响数值，但会让读者误以为 hybrid prefill 像 standalone CSA/HCA 一样包含 O(T) Python 循环，实际上 hybrid 的 cache build 是 O(n_blocks) Python 操作，常数小得多。
- **影响**：图/表的 caption 文字描述与 hybrid 实际实现不符，可能导致对 hybrid prefill 代价的误判（读者会以为 hybrid prefill 比实际更慢）。
- **修复**：对 hybrid 行 JSON 写 `prefill_cache_build = "batched_populate_from_naive_output"`，并在 make_figures 标题里用不同标注区分。

### P1-3　Exp4/Exp5 `conclusions_valid` 判定对"near_chance"的阈值是 `< 1.5 * chance`，太宽松，会把显著低于 chance 的"反学习"结果（0-4% 区间，在 vocab=16 chance=6.25% 下）误判为"near_chance"从而保护了失败的算子

- **涉及文件**：`run_quality.py::main()`（约 L1310）、`run_ablation.py::main()`（约 L655）
- **代码**：
  ```python
  near_chance = [r for r in all_results
                 if 'error' not in r
                 and r.get('mean_acc') is not None
                 and r['mean_acc'] < 1.5 * chance]     # 1.5 * 0.0625 = 9.375%
  conclusions_valid = (n_seeds >= 5 and min_seeds_ok >= 5
                       and n_any_sig > 0
                       and len(near_chance) < len(all_results) // 2)
  ```
- **问题**：
  - `mean_acc ≈ 0` 或 `2%`（显著低于 6.25% chance，即"反学习"或"未收敛"）同样满足 `< 9.375%`，被归类为"near chance"，但它们其实是**模型出了 bug 或训练未收敛**的强信号，不应该被 "near chance" 轻轻带过；
  - 第二个条件 `len(near_chance) < len(all_results) // 2` 意味着只要有 ≥一半算子在 0%-9.375% 区间就报 invalid——这对 KDA/CSA/HCA 都在 5-8% 震荡时很敏感，但对"全部反学习（0-3%）"不敏感；
  - 更严重的是：当所有种子都给出 ≈0% 的 mean_acc，t_stat 会是**很大的负值**，`significant_bonferroni = t_stat > crit` 会正确地返回 False，这是对的；但此时 `std_acc` 若接近 0（所有种子都 0/16=0），`t_stat = (0 - 1/16)/(0/√n) = -inf`，代码里 `if n > 1 and std_acc > 0` 会返回 None，保护了除零。但如果有一个种子蒙对一个 batch（比如 1/64 ≈ 1.5%），std 很小但 >0，会得到一个极大的负 t_stat，被正确判为不显著——这部分 OK。
- **影响**：不会改变数值，但会让**未收敛/反学习的算子行**在结论里看起来"只是 near chance"，而不是"训练 recipe 或算子实现有问题"。
- **修复**：将 near_chance 改为区间 `chance * 0.5 < acc < chance * 1.5`，并把 `< chance * 0.5` 的行归为 `far_below_chance`，在 summary 里单独计数；只要有任何一个 far_below_chance 行，conclusions_valid 就应该强制为 False（表示配置/训练有问题）。

## 三、P2 级 — 可复现性 / 边缘场景

### P2-1　`_make_op_gen` 在 CUDA 旧 torch 下 `torch.Generator(device='cuda')` 会抛错，但代码里没有 fallback

- **涉及文件**：`run_benchmark.py::_make_op_gen`（L78）、`run_decoding.py` 类似 generator
- **代码**：
  ```python
  return torch.Generator(device=device).manual_seed(name_hash ^ t_hash)
  ```
- **问题**：torch <2.0 不支持 `torch.Generator(device='cuda')`；虽然 pyproject 要求 `torch>=2.2,<2.7`，但在 2.2/2.3 的某些 CUDA wheel 上，CUDA generator 只在 CUDA 可用且编译时启用才支持（CPU-only wheel 会抛 RuntimeError）。此时整个 Exp2 直接崩溃，没有 fallback。
- **影响**：只在"装了 CUDA 版 torch 但 CUDA runtime 不可用"这类混合环境触发。
- **修复**：
  ```python
  try:
      g = torch.Generator(device=device)
  except Exception:
      g = torch.Generator()
  return g.manual_seed(name_hash ^ t_hash)
  ```
  并在 log 中声明 CPU generator。README 的 P2-2 已承认此问题但没有实际代码修复。

### P2-2　`conftest.py` 的 `--skip-cpu-slow` 逻辑使用 `pytest.skip` 在 module 收集阶段，而 run_correctness.py 中大量 slow 测试的 `@pytest.mark.slow` 装饰器没有配套的 `pytest_collection_modifyitems` 钩子在 `SKIP_SLOW=1` 时自动 skip——目前只靠 `run_all.py` 的环境变量截断 BENCH_LENGTHS/MQAR_STEPS，pytest 模式下 slow 测试仍会跑。

- **涉及文件**：`conftest.py`，`run_correctness.py`
- **影响**：只影响 pytest 开发者体验（用户用 `pytest run_correctness.py` 会跑全部 230+ 个测试包括长时正确性检查，预期 5-10 分钟但实际上可能半小时）。不影响实验结果。
- **修复**：在 `conftest.py` 增加
  ```python
  def pytest_collection_modifyitems(config, items):
      if config.getoption('--skip-slow', default=False) or os.environ.get('SKIP_SLOW') == '1':
          for item in items:
              if 'slow' in item.keywords:
                  item.add_marker(pytest.mark.skip(reason='SKIP_SLOW=1'))
  ```

### P2-3　`make_figures.py::fig_mqar`/`fig_ablation` 的 yerr 在 `ci95_acc is None` 时用 `or 0.0` 画成零长度 errorbar——这是对的（单 seed 或 n_seeds_ok=1 时不确定性无穷大，不画 errorbar 比画零误差条诚实），但**目前的实现 `cis.append(r.get('ci95_acc') or 0.0)` 会把 `ci95_acc=0`（n_seeds_ok=1 时返回的是 None，不会是 0；但将来若改成精确 0 也会被错误 coalesce）**——实际因为返回 None 不返回 0，不会触发；但 `chance = r.get('chance_acc') or chance` 会把 `chance_acc=0.0`（不合法，但防御写法）错误回退到默认——实际 chance=1/16=0.0625≠0，无影响。

- **影响**：纯代码健壮性问题，不影响当前输出。仅记录。

## 四、已核验无误的关键路径（未发现问题）

为便于复查，列出本次重点核验后确认**正确**的部分：

1. **CSA overlap 压缩**：`csa_compress_kv_overlapped` 与 `_csa_compress_kv_overlapped_single` 对第一块用 `-inf` pad，b-branch 的 softmax 权重为 0，输出退化为单分支压缩，因果方向正确；
2. **因果块 mask**：`_causal_block_mask`（`i_t[:, None] // m > i_b[None, :]`）→ 严格 `< t//m`；decoding cache 的 `b_threshold = t // m; cbm = arange(n_blocks) < b_threshold` 一致；
3. **Attention sink**：CSA/HCA/KV cache 三条路径都做了 `-row_max` 平移，logaddexp 稳定；
4. **KDA g_clamp_min**：naive / chunk / compiled 三条路径默认 `-10`，chunk 路径在 `cumsum` 之前 clamp，且 `g_diff.clamp(max=50)` 防 exp 溢出；
5. **naive_chunk_kda 的 Neumann 级数**：`(I-A)^{-1} A` 形式的 solve_triangular 正确；
6. **STE 方向**：`kv = kv + (soft_kv - soft_kv.detach())` 方向与文档一致（forward=hard，backward=hard+soft）；`full_softmax` 模式的 zero-in-forward aux 用 expand-同一张量而非 double-evaluate 已在 D11 修复；
7. **compiled_recurrent_kda 的 cache_key**：已包含 `repr(scale), repr(g_clamp_min), str(state_dtype), bool(initial_state is not None)`；
8. **torch 2.2 兼容性**：`_is_compiling_safely()` 已做 AttributeError 回退；
9. **KDA 非有限值检查**：`_warn_if_nonfinite` 被 `_is_compiling_safely()` 守卫，不会破坏 `fullgraph=True`；
10. **Sliding window chunked 路径**：`_sliding_window_attention` 按 chunk_t 切片 + unfold，数值 contract 与 unfold 等价（causal mask 通过 valid_mask 正确处理）；
11. **KV cache FLOPs 公式**：softmax_gqa 的 causal_entries `T*(T+1)//2`、CSA/HCA 的 `causal_block_entries` 三角+remainder 闭式、`effective_topk` 的三段式精确求和都已正确；输入投影 + grouped 输出投影已计入；
12. **MQAR RNG 隔离**：embed/head 先于 layer 创建，batch_gen 用 `seed + 1_000_000` 大偏移避免与 model-init RNG 重叠，eval_gen 固定 seed=12345；
13. **t 临界值**：`_bonferroni_crit_q` 使用 regularized incomplete beta + bisection，scipy 缺失时仍精确；`_t_crit_975` 对 n>100 委托给 `_bonferroni_crit_q`；
14. **one-sided t-test**：`t_stat > crit`（而非 abs），方向正确；Bonferroni 临界值用 `1-alpha` 上尾分位数；
15. **JSON 原子写入**：`write_json_atomic`（temp + fsync + os.replace）已在所有 run_*.py 中使用；
16. **naive_csa fuse_projections**：对 `use_ste=False` 的 zero-grad vs None-grad 差异有文档与测试覆盖；
17. **HybridConfig 校验**：GVA 整除、dropout 非 finite/类型、维度 ≥1、n_k/n_c/n_h 非负且不全为 0 等都已覆盖；
18. **CSA decode cache 的 _SlidingWindowRingBuffer**：wrap-around 逻辑、`get()` 正确处理 tail+head 拼接；
19. **Decoding softmax 的 KV cache**：`_ensure_cache_capacity` 几何增长 + O(1) 写入，无 O(T²) 拷贝；
20. **`short_conv` lookback**：left-pad by (ksize-1)、`detach_lookback` 逻辑已在 `KDAHybridLayer.forward_functional` 正确实现，跨 chunk BPTT 路径可走通。
