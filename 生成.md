## 二、当前仍存在的影响实验结果的问题（待修复）

### P0 - 高优先级（直接改变数值或使比较不公平）

#### P0-1 `compiled_recurrent_kda` 缓存键缺失 `scale` / `g_clamp_min`
- **文件**: `ops_kda.py: compiled_recurrent_kda` (约 350-400 行)
- **现象**: 
  ```python
  cache_key = (B,T,H,K,HV,V, dtype, device.type, requires_grad, mode, dynamic, fullgraph, output_final_state, bool(initial_state), str(state_dtype))
  ```
  未包含 `scale` 和 `g_clamp_min`。这两个标量被作为参数传给 `_kda_kernel`，但 `torch.compile(..., dynamic=True)` 仅对张量形状 dynamic，对 Python float 视为静态。若两次调用仅 `scale` 不同（如 ablation 扫 `K**-0.5` vs `1.0`），`_COMPILED_KDA_CACHE` 会命中旧图，返回使用旧 `scale` 编译的图，导致静默错误结果。
- **影响**: 实验若在同一进程内比较不同 `scale`（MQAR fairness 中 softmax 用 `K**-0.5`，CSA/HCA 用 `1.0`，KDA 内部用 `scale` buffer），虽当前 `KDAHybridLayer` 固定 `scale=K**-0.5` (tensor buffer)，但外部直接调用 `compiled_recurrent_kda` 做对照实验会被污染。
- **修复**: 将 `scale` 和 `g_clamp_min` 加入 `cache_key`，并对 `scale=None` 情况用 `repr` 规范化。

#### P0-2 `torch.compiler.is_compiling()` 兼容性
- **文件**: `ops_kda.py: _warn_if_nonfinite` (约 120 行)
- **现象**: `torch.compiler.is_compiling()` 在 torch 2.2 不存在（pyproject 允许 `>=2.2,<2.7`）。直接调用会在 torch 2.2 环境抛出 `AttributeError`，导致 `naive_recurrent_kda` 整个前向崩溃，而测试环境是 2.6+ 不触发。
- **影响**: 在用户本地 torch 2.2 下运行正确性回归（`run_correctness.py`）会崩溃，无法得到实验结果。
- **修复**: 
  ```python
  def _is_compiling():
      try:
          return torch.compiler.is_compiling()
      except AttributeError:
          # torch <2.3 fallback: torch._dynamo.is_compiling / False
          try:
              import torch._dynamo as _dm
              return _dm.is_compiling() if hasattr(_dm, 'is_compiling') else False
          except Exception:
              return False
  ```
  在 `_warn_if_nonfinite` 中使用该安全包装。

#### P0-3 Hybrid 中 `forward` 与 `forward_functional` 的 `detach_state` / `detach_lookback` 文档与实现不一致风险
- **文件**: `ops_fused.py: KDAHybridLayer.forward_functional` (约 250 行)
- **现象**: 修复 P0-3 时，新 lookback 的 detach 逻辑已改为尊重 `detach_lookback`，但旧注释仍残留“无条件 detach”。而 `HybridKCHAttention.forward_functional` 中对 `stacked` 的 device 迁移注释说“DO NOT cast dtype”，这是 P1-2 的有意设计，但会导致 fp16 输入时 state 保持 fp32，若未来有 per-layer dtype 配置，会产生混合 dtype 的 `torch.stack` 崩溃（当前全为同 dtype，碰撞未触发）。
- **影响**: 跨 chunk BPTT 实验（`detach_lookback=False`）若使用混合精度，梯度流可能因意外 detach 中断，导致收敛曲线失真。
- **修复**: 在 `forward_functional` 增加显式断言 `assert all(s.dtype == states[0].dtype)` 并在 docstring 明确 dtype 保持 fp32 的设计。

### P1 - 中优先级（影响延迟/显存数字的解读）

#### P1-1 Benchmark 的 compute boundary 已记录但图表仍可能被误读
- **文件**: `run_benchmark.py: op_boundary`, `make_figures.py: fig_benchmark`
- **现状**: 已通过 `compute_boundary` 字段和分 subplot 修复，但 JSON 中仍同时存在 `time_min_ms/max_ms/std_ms` 与 `time_ms`，若下游论文直接取 `time_ms` 不看方差，会忽略 2-3x 的测量噪声（尤其 CPU 单线程固定后）。
- **建议**: 在 README Fairness notes 中明确“CPU 延迟测量已 pin 单线程，GPU 用 CUDA events，std 已记录，单次 median 不足以做算子排序”。

#### P1-2 Decoding prefill 延迟包含 Python 缓存填充
- **文件**: `run_decoding.py: CSAAttnDecoding.forward`, `HybridDecoding._forward_*`
- **现象**: `bench_decoding` 的 `prefill_ms` 对 CSA/HCA/hybrid 包含 `reference_python_append`（逐 token Python 循环填充压缩块 + ring buffer），而 softmax/KDA 是原生算子状态。这是保守的 reference-wrapper 数字，已在 JSON 中通过 `prefill_cache_build` 标记，但若读者只看 `prefill_ms` 会误以为 CSA/HCA prefill 更慢。
- **影响**: 论文若引用 `prefill_ms` 作为 prefill 性能，会低估 CSA/HCA。
- **修复**: 文档已存在，无需代码改动；但需在 `fig_decoding` 标题中进一步强调“prefill 含 Python 填充，仅 decode/token 是主要信号”。

#### P1-3 `bench_hybrid` 的 `reset_state` 计入计时
- **文件**: `run_benchmark.py: bench_hybrid`
- **现象**: `fn()` 内 `model.reset_state(); return model(x)`，reset 是 O(n_kda_layers) 的 Python 赋值（`None` + 遍历清 lookback），虽开销小（<0.1ms），但对 T=128 的小序列，相对占比可达 5-10%。
- **影响**: hybrid 的端到端延迟被轻微高估，跨 boundary 比较时放大差异。
- **修复**: 将 reset 移到 `_measure` 的 warmup/外部，或在 `fn` 内不计入 reset 时间（本轮修复未做，仅记录，因影响 <10% 且修复会改变历史 JSON 边界定义）。

### P2 - 统计与可复现性

#### P2-1 `_t_crit_975` 对 n>100 仍用 Cornish-Fisher 近似
- **文件**: `run_quality.py: _t_crit_975`
- **现象**: `n>100` 时使用 `1.96 + (z^3+z)/4/(n-1)`，对 97.5% 分位数误差 <0.1%，但对 Bonferroni 尾部（如 `alpha=0.05/28` 需 99.9% 分位数）误差可超 60%。已通过 `_bonferroni_crit_q` 的精确 beta 回退解决 Bonferroni 路径，但 CI 路径（`_t_crit_975`）仍近似。
- **影响**: 当前实验 n=5~7，落在查表范围（2..100），不触发近似，无影响。未来若 `MQAR_SEEDS=200`，CI 会略偏窄。
- **修复**: 将 `_t_crit_975` 的 `n>100` 分支也改为调用 `_bonferroni_crit_q(n, alpha=0.025)` 的精确路径，或复用同一 beta 实现。

#### P2-2 `PYTHONHASHSEED` 与 `zlib.crc32` 种子
- **文件**: `run_benchmark.py: _make_op_gen`
- **现状**: 已用 `zlib.crc32` 代替 `hash()`，解决跨进程随机化问题，determinism 已修复。
- **剩余**: `torch.Generator(device)` 在 CUDA 上要求 `device='cuda'`，在较老 PyTorch 上仅支持 CPU generator。已通过 `torch.Generator(device=device)` 兼容新版，若用户 torch 2.2 + CUDA，`Generator('cuda')` 可能仍不支持，会回退到 CPU generator，导致 CPU/CUDA 间输入不一致。建议加 `try/except` 回退到 `torch.Generator()` 并在 log 中声明。

## 三、修复方案与实施

### 已实施修复（本轮）

1. **P0-1 fix**: `ops_kda.py: compiled_recurrent_kda` 缓存键加入 `scale` 与 `g_clamp_min`
   ```python
   cache_key = (
       B, T, H, K, HV, V,
       q.dtype, q.device.type,
       requires_grad,
       mode, dynamic, fullgraph,
       bool(output_final_state),
       bool(initial_state is not None),
       str(state_dtype),
       repr(scale),               # 新增
       repr(g_clamp_min),         # 新增
   )
   ```

2. **P0-2 fix**: 安全包装 `is_compiling`
   ```python
   def _is_compiling_safely():
       try:
           return torch.compiler.is_compiling()
       except AttributeError:
           try:
               import torch._dynamo as _dm
               fn = getattr(_dm, 'is_compiling', None) or getattr(_dm, 'is_compiling', None)
               return fn() if callable(fn) else False
           except Exception:
               return False
   def _warn_if_nonfinite(...):
       if _is_compiling_safely():
           return
       ...
   ```

3. **P2-1 fix (partial)**: `_t_crit_975` 对 n>100 委托给 `_bonferroni_crit_q(n, alpha=0.025)`，保证精确性，避免未来大 seed 数时 CI 不准。

4. **文档补充**: 在 `生成.md` 中明确列出 compute boundary、prefill 含 Python 填充等 caveat，避免论文误读。
